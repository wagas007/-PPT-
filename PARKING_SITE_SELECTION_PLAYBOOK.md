# 停車場案場選點 Playbook

> 目的：讓 AI 在執行停車場案場資料庫、試點選址、TOP10 初選、桃園 / 台中場站評估時，有一套可重用的技術檢查與商業判斷流程。  
> 適用：PARK24 / Times、城市車旅、嘟嘟房、ViVi PARK、台灣聯通、俥亭，以及後續所有停車場通路合作專案。

---

## 0. 使用時機

只要任務包含以下任一關鍵字，就必須先讀本文件：

```text
停車場選點
案場資料庫
候選場站
TOP10
桃園 / 台中場站
試點場站
供電分析
現勘清單
```

本文件只規範「選點方法與風險檢查」，不提供任何品牌可直接複製的合作文案。每個品牌的定位、敘事與合作主張仍必須來自該品牌的 `00_來源資料/`。

---

## 1. 核心原則

停車場選點不是找最多停車場，而是找最適合低干擾驗證合作模式的場站。

總經理層級通常會關心：

- 哪 1 至 3 個場站最適合先談？
- 為什麼是這幾個？
- 這些場站是否真實存在？
- 對方是否有權責協助評估？
- 是否不影響車位、動線與停車收益？
- 供電、補貨、客服、保險與法遵誰負責？
- 試點成功後是否具備擴大潛力？

因此案場資料庫最後必須能回答：

```text
存在性：這個場站是否在官方或可信來源中存在？
適配性：這個場站是否符合旺來試點條件？
可行性：供電、動線、邊角空間、權責是否可被確認？
敘事性：這個場站是否能支撐對方品牌願意合作的理由？
下一步：這個場站該現勘、保留、降級，還是排除？
```

---

## 2. 資料來源分級

案場資料必須分層，不要把事實與推論混在一起。

| 等級 | 來源類型 | 用途 |
|------|----------|------|
| A | 品牌官網案場查詢、官方新聞、官方招商頁、官方 PDF | 作為案場存在性的優先依據 |
| B | 政府公開資料、母公司公開資訊、正式新聞媒體 | 補強公司背景、權責與策略判斷 |
| C | Google Map、第三方停車平台、論壇或地圖評論 | 補強現場狀況，但不得單獨作為正式結論 |
| D | AI 推論、商業假設、人工判斷 | 只能放在推論欄位，不可寫成事實 |

每個案場至少應保留：

- `官方案場名稱`
- `官方地址`
- `資料來源`
- `資料來源等級`
- `是否為推論`
- `待現勘確認項目`

## 3. 官網技術檢查流程

停車場官網常見問題是：前台看得到搜尋頁，但案場資料不是直接寫在頁面文字中。

AI 不可只因頁面顯示 Loading、搜尋不到文字、或頁面需要互動，就判定找不到資料。

必須依序檢查：

```text
1. 開啟官方停車場查詢頁
2. 檢查 HTML 是否有案場 JSON 或 JavaScript 陣列
3. 搜尋常見關鍵字：parks、parking、allParks、store、lat、lng、address、county、city
4. 檢查 Network / API endpoint 是否存在
5. 檢查是否有城市篩選、分頁或關鍵字查詢限制
6. 若資料藏在 script 中，解析 script，不要只讀可視文字
7. 若仍無法取得，再補用官方新聞、Google Map 或第三方資料
```

### ViVi PARK 案例記錄

ViVi PARK 官方停車場頁面前台顯示 Loading，但 HTML 內有 JavaScript 變數 `allParks`，內含官方案場清單。  
因此下一次遇到類似頁面，應先檢查 script，而不是直接說找不到案場。

## 4. 官方存在不等於適合試點

官方清單上的場站可視為「存在性較可信」，但不代表適合設置。

以下狀況都需要降級或標記待確認：

- 地下室或室內場站，設備放置、補貨與安全性較複雜
- 車道緊、動線短、沒有明顯邊角空間
- 場站可能是特約、聯盟或委外，權責未必在該品牌
- 零售場站可能牽涉賣場、地主、物業多方同意
- 高鐵、車站或商場場站人流高，但管理與法規門檻也高
- 地址存在，但官網營運狀態可能已更新或暫停

資料庫中建議分成：

```text
官方存在
初步適合
待現勘確認
不建議優先
```

## 5. 標準評估欄位

建立案場資料庫時，至少應包含下列欄位：

| 欄位 | 說明 |
|------|------|
| 案場 ID | 內部統一編碼 |
| 官方案場名稱 | 以品牌官網為準 |
| 縣市 / 行政區 | 桃園、台中優先 |
| 官方地址 | 以來源原文保留 |
| 場站型態 | 室外平面、地下室、零售賣場、轉運節點、住宅周邊等 |
| 空間訊號 | 是否有旁、對面、路口、空地、平面等訊號 |
| 生活服務密度 | 周邊住宅、商圈、學校、辦公或日常採買 |
| 供電可行性 | 初步推論，仍需現勘 |
| 補貨維運可行性 | 車輛停靠、補貨路徑、工作時間 |
| 權責複雜度 | 自營、特約、聯盟、賣場、物業等 |
| 合作價值 | 對對方品牌的商業或形象價值 |
| 導入難度 | 低、中、高 |
| 優先級 | A、B+、B、C |
| 下一步 | 現勘、窗口確認、保留、排除 |
| 資料來源 | URL / 文件 / 人工實查 |
| 假設備註 | 把推論寫清楚 |

## 6. 麥肯錫式評估模型

選點時建議用二維矩陣：

```text
X 軸：導入難度
Y 軸：合作價值
```

優先級定義：

| 優先級 | 判斷 |
|--------|------|
| A | 合作價值高、導入難度低，適合列為首波現勘 |
| B+ | 合作價值高，但權責或現場條件需確認 |
| B | 可作為備選，但不適合第一波主打 |
| C | 存在但不建議優先投入 |

評估時避免只看熱鬧程度。大型商場、車站、高鐵場站雖然曝光高，但不一定是第一波最佳試點，因為權責、動線、消防、供電與管理成本可能更高。

第一波試點更適合：

- 室外平面
- 住宅或生活圈周邊
- 有邊角或空地訊號
- 補貨動線相對單純
- 對方容易內部評估
- 成功後具備同型場站複製性

## 7. 不同品牌不可共用結論

可以共用的是工具與欄位，不可以共用的是品牌敘事與合作主張。

```text
共用流程，不共用結論。
共用欄位，不共用策略敘事。
```

例：

- PARK24 / Times：重點可能是總部型合作與示範點可複製性
- 城市車旅：重點可能是台中在地場站、標準化管理、區域示範
- 嘟嘟房：重點可能是 Pre-IPO、智慧場域案例與品牌升級
- ViVi PARK：重點可能是 APP、Vi 幣、生活聯盟、會員點數與停車生活服務生態

新品牌必須重新讀該品牌的 `00_來源資料/`，再決定評估權重。

## 8. 必要輸出

案場資料庫完成後，應至少輸出：

- `台中場站庫`
- `桃園場站庫`
- `評分模型`
- `TOP10 初選`
- `TOP3 建議試點`
- `供電分析`
- `風險登錄`
- `行動計畫`
- `Source_Assumptions`

回報使用者時，必須清楚區分：

```text
哪些是官方事實？
哪些是 AI 推論？
哪些需要現勘？
哪些不建議優先？
```

---

## 9. 品質檢查

完成案場資料庫前，逐項檢查：

- 是否先讀當前品牌 `00_來源資料/`？
- 是否避免引用其他品牌的案場與文案？
- 是否檢查官方頁面是否有 script / API 隱藏資料？
- 是否保留官方名稱與官方地址？
- 是否標註資料來源等級？
- 是否區分事實與推論？
- 是否優先桃園 / 台中？
- 是否有 TOP10 與 TOP3？
- 是否有供電、動線、權責、現勘風險？
- 是否用總經理能決策的方式回報？

## 10. 對外說法原則

對外信件、OnePager、PPT 不要把案場資料庫的內部推論直接端出去。

對外應說：

```text
我們建議先以 1 至 3 個示範場站共同評估，
實際是否適合仍以貴司權責、供電、動線與現勘結果為準。
```

避免說：

```text
我們已選定你們的某某場站。
這幾個場站一定適合。
你們只需要提供場地。
```

對大型合作對象，要用共同評估、低干擾、可驗證、可回顧、可選擇是否擴大的語氣。
